## IMPORTANT FINDINGS (2)

### I-2 — A worker-thread panic wedges the UI forever (`poll()` discards `Disconnected`)

**File:** `src/gui/app.rs:142-144` (root cause) — trigger reachable via `src/archive/rpa.rs:492` → `src/archive/rpa.rs:107`
**Severity:** important — on a reachable panic the app becomes permanently unusable (spinner spins, all buttons stay disabled, no error shown) until the process is killed.

`poll()` collapses every channel error to `None`:

```rust
// src/gui/app.rs:141-145
let done = if let Some(rx) = &self.rx {
    rx.try_recv().ok()          // Empty AND Disconnected both become None
} else {
    None
};
```

`start()` spawns the job on a detached `std::thread`; the only way results return is `tx.send(...)`. If the worker **panics** instead of returning, `tx` is dropped without sending. `try_recv()` then yields `Err(TryRecvError::Disconnected)`, which `.ok()` turns into `None` — so `self.running` is never cleared. The toolbar is gated on `!busy` (`app.rs:186`, `add_enabled_ui`), so it stays disabled; the spinner (`app.rs` toolbar) never stops. Permanent wedge.

This is not hypothetical. The archive index parser builds entry lengths straight from the (corrupt-controllable) pickled index:

```rust
// src/archive/rpa.rs:492
length: Length::new(len),
// src/archive/rpa.rs:106-108
pub fn new(value: u64) -> Self {
    assert!(value > 0, "Length must be > 0; got {value}");   // panics on len == 0
```

A malformed/hostile `.rpa` whose index declares a zero-length entry makes `list_rpa`/`extract_rpa` **panic** (not return `Err`). In the GUI these run on the worker thread (`Job::Scan` calls `list_rpa` at `ops.rs:122`; `Job::Extract` calls `extract_rpa` at `ops.rs:183`), so the panic unwinds the worker and wedges the window.

**Reproducer:**
```
1. Build a .rpa whose zlib-pickled index contains an entry with length 0
   (Ren'Py RPA-3.0: [offset, 0, ""] triple), place it in a source dir.
2. Point the GUI Source at that dir, click Scan (or Extract with "Unpack .rpa").
3. Worker thread panics in Length::new(0); UI shows the spinner forever,
   Scan/Extract/Verify/Convert stay greyed out. Only killing the app recovers.
```
(Library-side sanity: `rpa.rs:605-608` already proves `Length::new(0)` panics via `catch_unwind` in a unit test — the panic is a known, exercised path.)

**Fix (GUI-side, sufficient on its own):** handle `Disconnected` explicitly so a dead worker is surfaced and clears `running`.

```rust
fn poll(&mut self) {
    use std::sync::mpsc::TryRecvError;
    let msg = self.rx.as_ref().map(|rx| rx.try_recv());
    match msg {
        Some(Ok(result)) => {
            self.rx = None;
            self.running = None;
            match result.outcome {
                Ok(log) => { self.push_log(log.trim_end()); self.status = "done".into(); }
                Err(e)  => { self.push_log(&format!("ERROR: {e}")); self.status = format!("error: {e}"); }
            }
        }
        Some(Err(TryRecvError::Disconnected)) => {
            self.rx = None;
            self.running = None;
            self.push_log("ERROR: worker terminated unexpectedly (panic)");
            self.status = "error: worker terminated".into();
        }
        Some(Err(TryRecvError::Empty)) | None => {} // still running / idle
    }
}
```
**Optional hardening:** wrap the body of the closure in `start()` with `std::panic::catch_unwind(...)` to convert a panic into a normal `Err` message. Cross-reference (out of GUI scope): `Length::new` panicking on attacker-controlled input is itself a library robustness smell — a corrupt archive should yield `RenpyExError`, not `panic!`.

### I-1 — `start()` documents an empty-path no-op guard that isn't implemented

**File:** `src/gui/app.rs:120-127`
**Severity:** important — missing input validation and a false documented contract; clicking an action with an empty Source/Output spawns a doomed worker that fails with a raw filesystem error instead of the documented no-op.

```rust
/// Spawn a background worker for `job`. No-op if one is already running or
/// required paths are empty.                         // <-- promise
fn start(&mut self, job: Job) {
    if self.running.is_some() {
        return;
    }
    let source = PathBuf::from(self.source.trim());    // no empty check
    let output = PathBuf::from(self.output.trim());    // no empty check
    ...
```

The "required paths are empty" clause is never enforced. With an empty Source, `Job::Scan`/`Extract`/`Verify` reach `fs::read_dir("")` / `verify_all` and error; with an empty Output, `Extract`/`Convert` reach `prepare_output(Path::new(""), ..)` and error. No crash, but the user gets a cryptic `ERROR: ... (os error ...)` in the log rather than the intended quiet no-op, and the toolbar is not gated on path presence.

**Reproducer:**
```
1. Launch renpyex-gui with a fresh config (both path fields blank).
2. Click "Scan".
3. Observe a raw OS error in the log instead of nothing happening.
```

**Fix:** implement the documented guard (and disable the buttons when paths are missing).

```rust
fn start(&mut self, job: Job) {
    if self.running.is_some() { return; }
    let source = self.source.trim();
    if source.is_empty() { self.status = "set a Source directory first".into(); return; }
    let needs_out = matches!(job, Job::Extract | Job::Convert);
    if needs_out && self.output.trim().is_empty() {
        self.status = "set an Output directory first".into();
        return;
    }
    let source = PathBuf::from(source);
    let output = PathBuf::from(self.output.trim());
    ...
```
Optionally gate the toolbar buttons: `add_enabled_ui(!busy && !self.source.trim().is_empty(), ...)`.

## MINOR FINDINGS (4)

### M-1 — Long log lines are clipped with no horizontal scroll

**File:** `src/gui/app.rs:254-266`, builder at `src/gui/app.rs:375-393`
**Severity:** minor — readability regression vs. the previous `TextEdit`; failure lines with long absolute Windows paths get cut off at the right edge.

`colorize_log` builds a `LayoutJob` with default wrapping (`TextWrapping::default()` → `max_width = f32::INFINITY`, i.e. no wrap). It is rendered inside `ScrollArea::vertical()` (`app.rs:254`), which constrains width to the panel and **clips** horizontal overflow — there is no horizontal scrollbar. The prior `TextEdit::multiline` wrapped, so nothing was hidden. Lines like `  convert fail C:\very\long\path\...: <error>` will be truncated and unreadable.

**Reproducer:**
```
Run an Extract that fails on a file with a long absolute path; the tail of the
failure line runs past the right edge with no way to scroll to it.
```

**Fix:** allow horizontal scrolling (keeps monospace log alignment), or wrap.

```rust
egui::ScrollArea::both()               // was ::vertical()
    .auto_shrink([false, false])
    .stick_to_bottom(true)
    .show(ui, |ui| { ... });
```

### M-3 — `colorize_log` rebuilds the entire `LayoutJob` every frame during a job

**File:** `src/gui/app.rs:266` (called from `update`), repaint pump at `src/gui/app.rs:275`
**Severity:** minor — wasted per-frame work; low impact for typical logs, grows with log size.

While a job runs, `update()` calls `ctx.request_repaint()` unconditionally (`app.rs:275`), so `colorize_log(&self.log)` re-scans the whole accumulated log and re-appends it segment-by-segment into a fresh `LayoutJob` on **every** frame (~60/s), even though `self.log` only changes when `poll()` drains a result. egui's galley cache avoids re-*layout* of an identical job, but the `LayoutJob` construction itself is redundant.

**Fix:** cache the built job and rebuild only when the log changes.

```rust
// in RenpyExApp: cached: Option<(usize /*log len*/, egui::text::LayoutJob)>
let job = match &self.cached {
    Some((len, j)) if *len == self.log.len() => j.clone(),
    _ => { let j = colorize_log(&self.log); self.cached = Some((self.log.len(), j.clone())); j }
};
ui.add(egui::Label::new(job).selectable(true));
```
(Or rebuild in `poll()` only, since that is the only writer of `self.log`.)

### M-2 — Redundant `#[cfg(feature = "gui")]` gates inside an already-gated module

**File:** `src/gui/app.rs:20, 181, 280, 336, 350, 374, 394` (and the `use crate::gui::theme;` at `:20`)
**Severity:** minor — dead attributes; misleading, implies the file can build without the feature.

`lib.rs:19-20` declares `#[cfg(feature = "gui")] pub mod gui;`, so the entire `gui` module (and thus all of `app.rs`) only compiles when the feature is on. The inner `#[cfg(feature = "gui")]` on the `theme` import, the `impl eframe::App`, and the four helper fns are therefore always-true and never toggle anything.

**Reproducer:** static reasoning — `cargo build` (no features) never compiles `app.rs`, so the inner gates can only ever be active; `cargo build --features gui` compiles them unconditionally. No configuration makes them evaluate to false.

**Fix:** drop the inner `#[cfg(feature = "gui")]` attributes in `app.rs` (keep the single gate in `lib.rs`). Harmless as-is, but removing them prevents the false impression that `app.rs` is partially feature-independent.

## Root-cause pattern

Both important findings share one root: **the worker/UI boundary is optimistic** — it assumes the worker always returns a `JobResult` and always receives valid paths. `poll()` treats "no message" and "channel dead" as identical, and `start()` trusts the path fields. Neither is a logic error in the happy path (which is why tests pass), but both fail the moment reality deviates: a panic, or an empty field.

**Recommended first action:** fix I-2 (`poll()` handling of `Disconnected`) — it is three lines and converts a permanent, restart-only hang into a visible error.

**Order of operations:**
1. I-2 — `poll()` `Disconnected` arm (unblocks the worst UX failure; independent).
2. I-1 — empty-path guard in `start()` + optional button gating (independent).
3. M-1 — `ScrollArea::both()` (one-line readability fix).
4. M-3 / M-2 / M-4 — polish; do together in one pass.

_All findings are in `src/gui/app.rs`; `ops.rs`, `config.rs`, `theme.rs`, `mod.rs`, `bin/gui.rs`, `Cargo.toml`, and `gui_smoke.rs` are clean._
